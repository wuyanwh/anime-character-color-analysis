## 一、案例主题

本次课程项目的主题是“动漫角色发色瞳色与角色特点关联分析”。我主要围绕动漫角色的外观特征展开分析，把发色、瞳色和萌点等信息整理成结构化数据，再通过数据分析和图像识别方法，观察不同外观设定和角色特点之间是否存在一定规律。

这个选题一开始看起来比较偏兴趣向，但在实际完成过程中，我发现它包含了比较完整的数据处理流程：数据采集、数据清洗、特征标准化、可视化分析、聚类分析、关联规则挖掘，以及最后的 Web 展示。它不是单纯做一个静态展示，而是需要把数据分析结果和实际应用结合起来。

## 二、项目完成过程

在数据准备阶段，我们参考了世萌比赛角色和萌娘百科等资料来源，整理角色姓名、作品、发色、瞳色和萌点信息。为了提高采集效率，我保留并整理了两份萌娘百科爬取脚本：`py.py` 主要通过页面标签定位“发色”“瞳色”“萌点”，`py2.py` 则通过角色信息表的 `tr/td` 结构定位字段。两份脚本都使用 Playwright 打开萌娘百科页面，从 `names.xlsx` 读取角色姓名，并把采集结果保存到 `results.xlsx`，后续再进行人工检查和清洗。

原始数据中存在很多不统一的写法，例如“金发”“金色”“blond”可能都表示类似颜色，所以需要先把发色和瞳色统一成标准类别。部分角色的萌点描述也比较杂，需要统一分隔符并去掉空值。萌娘百科页面结构也并不完全一致，有些角色页面使用信息表，有些字段位置会变化，因此两份爬虫脚本分别采用不同提取方式，便于在页面结构变化时互相补充。

在数据分析阶段，我学习并使用了 Pandas 对数据进行清洗和统计，通过发色分布、瞳色分布、发色和瞳色交叉关系等方式观察整体规律。比如黑发和棕瞳比较常见，金发和蓝瞳也经常一起出现，而红发、紫发等颜色更容易和热血、神秘等角色印象联系起来。

在模型部分，项目尝试了传统 OpenCV 检测、YOLO 思路和最终的 OpenCV + K-Means 方案。由于动漫角色图片和真实人脸差异较大，直接使用普通人脸检测模型效果不稳定；而 YOLO 方案对标注数据量要求较高，在课程项目时间内不太容易训练出可靠模型。最后采用区域估算、HSV 颜色空间筛选和 K-Means 聚类提取主色，是比较适合本项目规模的方案。

在应用展示阶段，我把原来的多个脚本整理成了更清楚的项目结构。Web 端可以上传角色图片，程序会提取头发和眼睛区域的主色，并输出识别到的发色和瞳色。如果提供角色数据库，还可以根据识别结果推荐相似角色，并统计共同萌点。

在项目整理阶段，我也把原来分散在根目录的萌娘百科采集代码放入 `src/crawlers/` 文件夹，并在 README 中补充了运行方式、输入输出文件和注意事项。这样项目结构中可以同时体现数据采集、数据分析和 Web 展示三个部分，而不是只保留最终分析代码。

## 三、个人收获

通过这个案例，我对“用户数据采集与关联分析”的完整流程有了更直观的理解。以前我对数据分析的理解主要停留在统计表格和画图，这次项目让我认识到，真正的数据项目往往需要先解决数据来源不统一、字段缺失、类别标准不一致等问题。数据清洗虽然不如建模看起来复杂，但它会直接影响后续结果是否可信。

我也进一步理解了 K-Means 和 Apriori 的实际用途。K-Means 可以把发色、瞳色编码后的角色划分成若干相似群体，帮助观察角色外观设定的聚集情况；Apriori 则可以从事务数据中挖掘“发色/瞳色 -> 萌点”的关系，用支持度、置信度和提升度衡量规则强弱。这些方法放在具体案例中，比单独学习公式更容易理解。

另外，图像识别部分让我意识到模型效果和数据场景有很大关系。动漫图片存在夸张比例、高光、渐变色、遮挡和背景干扰，普通人脸识别方法不能直接套用。为了让结果更稳定，需要根据任务特点设计检测区域、过滤背景颜色，并对颜色进行聚类。

## 四、存在不足

本项目仍然存在一些不足。首先，角色数据库规模和字段质量会影响推荐效果，如果数据中萌点标签不够规范，关联规则结果也会受到影响。其次，图像识别部分对图片构图比较敏感，如果角色不是正脸，或者头发、眼睛被遮挡，识别结果可能不准确。再次，当前颜色分类仍然是规则映射，面对渐变发色、双色瞳、特殊滤镜时还有改进空间。

数据采集方面也存在局限。萌娘百科页面可能触发反爬限制，不同角色页面的信息表结构也不完全统一，脚本采集到的“萌点”有时会混入页面链接或别名信息。因此采集结果不能直接作为最终数据使用，仍然需要人工抽查、字段校正和类别归一化。

此外，项目中的推荐功能主要依据发色、瞳色和萌点做简单匹配，还没有加入作品类型、角色定位、人气等更多维度。如果后续继续完善，可以把更多特征加入模型中，让推荐结果更有解释性。

## 五、改进方向

后续可以从三个方面继续优化。第一，扩充和规范角色数据库，增加更多角色样本，并统一发色、瞳色和萌点标签。第二，引入更适合动漫图像的检测模型，例如专门的动漫人脸检测模型或多标签分类模型，提高发色和瞳色识别准确率。第三，完善 Web 展示页面，加入可视化图表、历史记录和规则解释，让用户更容易理解推荐依据。

总体来说，这次案例让我把课堂中的数据采集、清洗、建模、关联分析和系统展示串联了起来。虽然项目还不够完善，但它让我体会到一个数据分析项目从想法到可运行结果的完整过程，也让我认识到数据质量、模型选择和应用场景之间需要相互配合。
